# sub2-monitor 平台需求说明

## 1. 背景

本系统用于监控 `sub2api` / `newapi` 两类服务平台的账号消耗、余额、分组倍率变化，并支持定时采集、告警通知、历史记录留存和可视化展示。

## 2. 技术栈

- 数据库：`PostgreSQL`
- 后端：`Spring Boot 3`、`JDK 21`、`MyBatis-Plus`、`Spring AI`
- 前端：`Vue 3`、`Shadcn-vue`

## 3. 目标

1. 统一管理多个服务平台、多个站点、多个账号。
2. 定时采集余额、分组、倍率、账号状态等信息。
3. 支持邮件汇总告警，避免每次单项变化都单独发信。
4. 支持 SMTP 配置管理与历史邮件查看。
5. 支持历史数据独立入库、折线图展示和审计记录。
6. 支持根据采集结果自动调整账号权重或禁用账号。

## 4. 范围说明

### 4.1 平台范围

当前仅支持两类平台：

- `newapi`
- `sub2api`

平台下可存在多个站点，每个站点可存在多个账号。

### 4.3 模型平台枚举

账号所属模型平台枚举值：

- `openai`
- `anthropic`
- `gemini`
- `antigravity`

### 4.2 账号范围

系统仅管理本平台可操作的账号，不直接修改第三方平台未纳入管理范围的账号。

## 5. 核心功能

### 5.1 平台管理

#### 功能

- 新增平台
- 编辑平台
- 查看平台列表
- 启用/停用平台

#### 平台字段

- `base_url`：平台接口地址
- `name`：平台名称
- `billing_ratio`：充值到账号的比例
- `top_up_ratio`：是否需要手动控制的充值比例规则
- `remark`：备注

#### 规则

- `base_url` 必填且唯一。
- 平台名称需可读、可搜索。
- 充值比例支持不同平台不同配置。
- 同一平台下可对应多个账号。

### 5.2 账号管理

#### 功能

- 新增账号
- 编辑账号
- 删除账号
- 启用/禁用账号
- 查看账号详情
- 修改账号权重
- 绑定代理

#### 展示规则

- 账号名称正常回显。
- 密码或敏感密钥不做明文回显。
- 支持展示账号状态、最近使用时间、是否可调度、限流状态、过载状态、临时不可调度原因。

#### 字段说明

- `name`：账号名称
- `platform`：所属平台
- `type`：账号类型
- `credentials`：账号凭证信息
- `extra`：扩展信息
- `proxy_id`：代理配置
- `concurrency`：并发数
- `priority`：优先级
- `status`：账号状态
- `schedulable`：是否允许参与调度
- `rate_limited_at`：限流发生时间
- `rate_limit_reset_at`：限流恢复时间
- `overload_until`：过载解除时间
- `temp_unschedulable_until`：临时不可调度解除时间
- `temp_unschedulable_reason`：临时不可调度原因
- `expires_at`：过期时间
- `auto_pause_on_expired`：过期是否自动暂停
- `rate_multiplier`：倍率
- `load_factor`：负载因子
- `notes`：管理员备注

#### 规则

- 账号启用/禁用应有审计日志。
- 账号是否可用以 `schedulable` 字段为准：`true` 表示可用，`false` 表示不可用。
- 权重修改需记录修改前后值与操作者。
- 账号过期后可根据配置自动暂停。
- 临时不可调度状态应有明确原因，便于排障。
- `type` 枚举值为 `apikey`、`Oauth`。
- `status` 枚举值为 `active`、`error`。
- `credentials` 为 JSON 结构，当前只需匹配包含 `api_key`、`base_url`、`pool_mode`、`model_mapping`、`pool_mode_retry_count` 的结构。

#### `credentials` 示例

```json
{
  "api_key": "sk-xxx",
  "base_url": "https://api.example.com/",
  "pool_mode": true,
  "model_mapping": {
    "gpt-5.4": "gpt-5.4-mini",
    "gpt-5.5": "gpt-5.5",
    "gpt-5.4-mini": "gpt-5.4-mini"
  },
  "pool_mode_retry_count": 3
}
```

### 5.3 定时采集

#### 采集内容

- 余额
- 分组
- 倍率
- 账号状态
- 账号连通性
- `token` 或 `llm token` 延迟
- 代理连通性

#### 执行方式

- cron 表达式需通过下拉选择或组件选择，避免手动输入错误。
- 每类任务需可单独配置启用/停用。
- 支持每个平台独立配置采集策略。

#### 重试策略

- 采集失败后需要重试。
- 重试次数、重试间隔、退避策略、超时时间均需配置化。
- 需明确失败后从哪个阶段继续重试。
- 重试配置应从配置表读取，不应写死在代码中。
- 默认重试次数为 3 次。
- 默认重试间隔为 5 秒。

#### 汇总发送

- 不允许每次单项变化单独发信。
- 应在所有平台采集完成后，汇总余额、分组、倍率变化后统一发送一封邮件。
- 汇总粒度为每轮采集一次。
- 如果某个平台任务提前失败，也需要等待全部任务结束后再统一发送。
- 若本轮无变化，则不发送邮件。
- 邮件内容按平台分组展示。

### 5.4 告警通知

#### 告警类型

- 余额变化告警
- 分组变化告警
- 倍率变化告警
- 账号异常告警
- 采集失败告警

#### 规则

- 各类告警是否启用需可配置。
- 支持按平台、账号、变化类型分别开关。
- 告警要支持合并、去重和失败补发。

### 5.5 SMTP 配置

#### 功能

- 配置 SMTP 服务信息
- 保存发件人信息
- 保存收件人配置
- 查看历史邮件记录

#### 规则

- SMTP 配置需可在页面上修改。
- 邮件发送日志需要保留。
- 历史邮件页需支持查看发送时间、主题、收件人、状态、失败原因。

### 5.6 历史数据

#### 要求

- 采集到的历史数据需单独入库。
- 需要初始化脚本自动建库、建表、初始化基础数据。
- 初始化脚本应支持重复执行或幂等执行。

#### 数据范围

- 余额历史
- 分组历史
- 倍率历史
- 告警历史
- 邮件历史
- 权重变更历史
- 历史数据保留 15 天。
- 超过保留期后物理删除。
- 邮件日志、告警日志、权重变更日志同样保留 15 天，超过保留期后物理删除。

### 5.7 折线图

#### 要求

- 展示若干关键指标的趋势变化。
- 支持按时间范围查看。
- 支持按平台、账号、站点过滤。
- 图表展示时间粒度根据定时任务 cron 表达式自动确定。

#### 口径

- 采样时间粒度需明确，例如：分钟、小时、天。
- 缺失值和异常值的展示方式需明确。
- 不同指标的坐标轴和单位需统一定义。

### 5.8 权重自动调整

#### 功能

- 所有数据采集完成后，可根据配置开关决定是否自动执行权重调整。
- 调整逻辑参考 `sub2api` 现有实现。
- 账号表中的 `weight` 或等效字段应支持直接更新。

#### 规则

- 调整前需先确认采集结果完整。
- 汇总采集完成后统一执行权重修改。
- 权重修改完全自动执行，不需要人工二次确认。
- 调整后需要写日志。
- 页面需展示修改依据与修改结果。

### 5.9 连接性检测

#### 功能

- 从本服务读取 `sub2api` 的 `accounts` 表信息。
- 读取 `base_url`、`api_key` 和代理信息。
- 向 `base_url` 发请求检测连通性。

#### 要求

- 该能力可使用 `Spring AI` 或普通 HTTP 客户端实现。
- 检测结果需要记录。
- 失败时记录具体原因与重试结果。
- 测试模型由页面配置。
- 连通性测试以完整响应成功作为通过标准。
- 测试失败 3 次后将账号 `schedulable` 修改为 `false`。
- 下次测试可用时将账号 `schedulable` 恢复为 `true`。

## 6. 数据库设计要求

### 6.1 `accounts` 表

`accounts` 表已包含以下关键字段：

- 基础信息：`name`、`platform`、`type`
- 凭证与扩展：`credentials`、`extra`
- 调度：`concurrency`、`priority`、`schedulable`
- 状态：`status`、`error_message`
- 限流与过载：`rate_limited_at`、`rate_limit_reset_at`、`overload_until`
- 临时不可调度：`temp_unschedulable_until`、`temp_unschedulable_reason`
- 生命周期：`created_at`、`updated_at`、`deleted_at`、`expires_at`
- 倍率：`rate_multiplier`
- 负载：`load_factor`

### 6.2 其他表

至少应包含：

- 平台表
- 站点表
- 代理表
- 采集任务表
- 采集历史表
- 邮件日志表
- 告警日志表
- 权重变更日志表
- 系统配置表

## 7. 权限要求

- 管理员可新增、编辑、删除平台和账号。
- 普通用户仅可查看被授权的数据。
- 权重修改、账号禁用、SMTP 配置修改应保留操作日志。
- 禁止越权修改未归属本平台的账号。

## 8. 日志与审计

- 记录采集开始、成功、失败、重试信息。
- 记录配置变更信息。
- 记录邮件发送记录。
- 记录权重变化依据与执行结果。
- 记录账号禁用/启用的操作人和时间。

## 9. 页面需求

### 9.1 平台管理页

- 平台列表
- 新增/编辑平台弹窗
- 平台状态展示

### 9.2 账号管理页

- 账号列表
- 账号详情
- 权重修改入口
- 启用/禁用入口
- 代理绑定入口

### 9.3 配置页

- 定时任务配置
- 告警开关配置
- SMTP 配置
- 采集重试配置

### 9.4 历史邮件页

- 邮件列表
- 邮件状态
- 邮件内容预览

### 9.5 折线图页

- 支持趋势图查看
- 支持筛选平台、账号、时间范围
